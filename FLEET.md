# 시옷앱 게이트웨이 — 양산·함대 관리 가이드

이 문서는 시옷앱 게이트웨이를 **대량 제조·배포·관리**하는 방법을 설명한다.  
개발·단일 기기 설치는 [siot-app/DOCS.md](siot-app/DOCS.md)와 [siot-app/DEPLOY.md](siot-app/DEPLOY.md)를 참고하라.

---

## 1. 게이트웨이 고유 ID

| 항목 | 내용 |
|---|---|
| **gatewayId** | 하드웨어 시리얼 기반 UUIDv5로 자동 생성 — 같은 기기면 항상 같고, 다른 기기면 항상 다르다 |
| **충돌 위험** | 없음. UUID 네임스페이스 특성상 시리얼이 다르면 UUID가 달라진다 |
| **사람이 읽는 시리얼** | 제조 시 라벨 부착. 플랫폼이 이 시리얼과 gatewayId를 매핑해 관리한다 |
| **저장 위치** | 애드온 `/data` 영속 볼륨(HA OS 보장) |

## 2. SSD 복제 양산

골든 이미지 1개를 복제해 대량 생산할 때의 전략이다.

### 문제: 복제 전 `/data`에 gatewayId·토큰이 남아 있으면?

| 해결 방법 | 설명 | 권장 여부 |
|---|---|---|
| **(a) 복제 전 `/data` 비우기** | 골든 이미지를 플래시하기 전에 `/data/siot-app/` 내용을 삭제 | 확실하나 수동 작업 추가 |
| **(b) 하드웨어 변경 자동 감지** | 시옷앱이 부팅 시 저장된 시리얼과 현재 하드웨어 시리얼을 비교해 불일치 감지 → gatewayId 재생성·기존 토큰 폐기·재등록 진행 (이미 구현됨) | **권장** |

방법 (b)를 쓰면 **그냥 복제해도 안전**하다. 각 기기가 첫 부팅 시 자신의 고유 ID를 자동으로 확정한다.

### 양산 흐름 요약

```
골든 이미지(SD/SSD) 제작
        ↓
  복수 매체로 dd/etcher 복제
        ↓
  각 기기 첫 부팅
        ↓
  시옷앱: 하드웨어 시리얼 감지 → UUIDv5 gatewayId 자동 생성
        ↓
  온라인 시 플랫폼에 자가 등록(claim API)
        ↓
  플랫폼 대시보드에 기기 표시
```

## 3. HA OS 복제 주의사항

HA OS는 자체적으로 다음 데이터를 디스크에 저장한다:

| 저장 항목 | 복제 시 문제 |
|---|---|
| machine-id / instance UUID | 두 기기가 동일 ID를 가져 플랫폼·네트워크에서 중복 충돌 |
| 온보딩 계정 (관리자 비밀번호 등) | 보안 위험 — 모든 복제본이 동일 계정 |
| `homeassistant.local` mDNS 이름 | 같은 망에 두 기기가 있으면 이름 충돌로 접속 불안정 |

### 권장 해결 방법

| 규모 | 권장 방식 |
|---|---|
| **소량(~10대)** | 복제 후 각 기기에서 HA 리셋(온보딩 재진행). 수동이지만 확실하다 |
| **대량(10대 이상)** | 스톡(미설정) HA OS만 복제 → 첫 부팅 후 자동 온보딩 → GitHub 저장소로 시옷앱 애드온 자동 설치(아래 §5 참고) |

대량 양산에서는 완전 온보딩된 HA OS 복제를 **피하는 것**이 원칙이다.

## 4. 함대 관리 모델

시옷앱 게이트웨이는 **자가식별·자가등록·상태보고**만 담당하고, 중앙 플랫폼이 나머지를 담당한다.

| 책임 | 게이트웨이(시옷앱) | 플랫폼(백엔드·대시보드) |
|---|---|---|
| 고유 ID 생성 | UUIDv5 자동 생성 | — |
| 등록(claim) | `POST /siot-app/api/v1/claim` 수신 | 코드 검증 후 토큰 발급 |
| 상태 보고 | API 응답 제공 | `GET /siot-app/api/v1/system` 폴링 |
| 기기 대장 | — | 전체 기기 목록·상태 관리 |
| OTA 트리거 | HA가 자동 감지 | 새 버전 GitHub push |
| 원격 진단 | 로그·상태 API 노출 | 대시보드에서 집계 |

### 상태 조회 API

플랫폼이 각 게이트웨이의 상태를 폴링할 때 사용하는 엔드포인트:

```
GET http://<게이트웨이-IP>:3000/siot-app/api/v1/system
Authorization: Bearer <gatewayToken>
```

응답에 버전·온라인 여부·HA 연결 상태 등이 포함된다.

## 5. OTA(소프트웨어 업데이트)

### 기본 경로: HA 애드온 저장소

1. 이 GitHub 저장소를 HA에 저장소로 등록한다(최초 1회).
2. `siot-app/config.yaml`의 `version`을 올린 뒤 `main` 브랜치에 push.
3. HA Supervisor가 새 버전을 감지하면 각 게이트웨이의 애드온 화면에 **"업데이트"** 버튼이 나타난다.
4. 사용자 또는 자동화가 버튼을 누르면 업데이트 적용.

### 빠른 경로: GHCR 프리빌드 이미지

온디바이스 빌드(Pi에서 직접 컴파일)는 5~15분이 걸린다. 프리빌드 이미지를 쓰면 이미지 pull만 하면 되므로 업데이트가 훨씬 빠르다.

**활성화 방법** (현재는 비활성 — 아래 줄 추가 전까지 온디바이스 빌드 유지):

1. GitHub Actions(`.github/workflows/build.yml`)가 GHCR에 이미지를 push하도록 저장소를 설정한다.
2. `siot-app/config.yaml`에 아래 줄을 추가한다:

   ```yaml
   image: ghcr.io/dyiot36524-prog/siot-app-{arch}
   ```

   (GHCR는 소문자만 허용 — `dyiot36524-prog`는 그대로 OK. `{arch}`는 HA Supervisor가 자동 치환.)

3. GHCR 패키지 공개 설정: GitHub → 패키지 → siot-app → 공개(Public)로 변경 (또는 HA 기기에 GHCR 인증 구성).

> ⚠️ `config.yaml`에 `image:` 필드를 추가하기 **전까지**는 현재 로컬(온디바이스) 빌드 방식이 그대로 유지된다. 기존 동작을 깨지 않으니 준비됐을 때 추가하면 된다.

## 6. 제조 흐름 요약

```
1. 골든 이미지 제작
   - 스톡 HA OS 설치 → 기본 설정만 완료
   - 시옷앱 저장소 등록(§아래)까지 자동화 가능

2. 대량 복제
   - dd / Raspberry Pi Imager 등으로 SSD/SD 복제
   - HA OS 미온보딩 상태 권장(기기별 고유 계정·ID 생성 유도)

3. 첫 부팅 (기기별 자동)
   - HA OS 온보딩(관리자 계정 생성)
   - 시옷앱 애드온 설치(저장소 등록 자동화 시 바로 설치 버튼)
   - 시옷앱 기동 → UUIDv5 gatewayId 자동 확정

4. 온라인 후 자가 등록
   - 시옷플랫폼에서 gatewayToken 발급(claim API)
   - 플랫폼 대시보드에 기기 표시

5. 운영
   - 플랫폼이 GET /system 으로 상태 폴링
   - 새 버전 push 시 HA 업데이트 버튼 → OTA
```

## 7. HA 저장소 등록 방법

1. 이 GitHub 저장소를 `main` 브랜치에 push한다(루트에 `repository.yaml` 포함).
2. 각 HA 기기에서: **설정 → 애드온 → 애드온 스토어 → ⋮ → 저장소**.
3. `https://github.com/dyiot36524-prog/siot-gateway` 입력 후 **추가**.
4. 애드온 스토어에 "시옷앱 게이트웨이"가 나타나면 **설치**.

> 대량 기기에서 저장소 등록을 자동화하려면 HA REST API(`POST /api/hassio/addons/repositories`)나 구성 자동화 도구를 활용한다.
